## 1. 미구현 항목 우선순위 (UX 가치 기준)

### 우선순위 1: 유저별 블로그 (`/blog/:username`)
- **UX 가치:** 매우 높음 — 멀티블로그 플랫폼 정체성 확립
- **비즈니스 임팩트:** 극대 — 사용자 체류시간, 커뮤니티 기초 형성
- **DB:** `blogs` 테이블 (향후), 현재는 `User.username` 기반

### 우선순위 2: 카테고리 + 태그 시스템
- **카테고리:** 한 블로그 내에서만 유효한 정리 체계 (📁 폴더)
- **태그:** 모든 블로그를 가로지르는 자유로운 발견 라벨 (🏷️ 라벨)
- **DB:** `categories`, `tags`, `post_tags` 테이블

### 우선순위 3: 포스트 공개 범위 설정
- **컬럼:** `posts.visibility` (public/members_only/private)
- **목적:** 개인정보 보호, 사용자 신뢰도 향상

### 우선순위 4~5: 예약 발행 + 검색 고도화
- 예약 발행: APScheduler 1분 간격
- 검색 고도화: Fulltext 인덱스 + 작성자/카테고리/태그 필터

## 2. Sprint 2 구현 순서 (사용자 가치 기준)

### ⚠️ 주의: "DB 기술 순서"가 아닌 "사용자 가치 순서"

**잘못된 순서:**
```
DB 스키마 → API CRUD → FE 페이지
→ 초반에 사용자가 데이터 없어서 가치 못 느낌
```

**올바른 순서:**
```
1단계: 블로그 홈 기초 인프라 (1주)
2단계: 태그 시스템 (2주)
3단계: 카테고리 시스템 (2주)
4단계: 블로그 홈 완성 (1주)
```

### Step 1: 유저별 블로그 기초 (1주)

**BE:**
- `GET /api/users/:username` — 사용자 정보 조회
- `GET /api/posts?author=:username` — 특정 사용자 포스트만

**FE:**
- `/blog/:username` 라우팅 + 기본 포스트 목록

**사용자 가치:**
```
"다른 블로그에 들어갈 수 있다"
→ 멀티블로그 정체성 확립 (카테고리/태그 없어도 됨)
```

---

### Step 2: 태그 시스템 (2주)

**DB:**
- `tags` 테이블 (blog_id, name, unique constraint)
- `post_tags` 테이블 (post_id, tag_id)

**BE:**
- POST/GET/DELETE `/api/tags` CRUD
- POST/DELETE `/api/posts/:id/tags` 태그 추가/제거

**FE:**
- 에디터 "태그 입력" 필드 (자동완성)
- 포스트 상세 "#태그" 표시 (클릭 가능)

**사용자 가치:**
```
"태그로 글을 모아볼 수 있다"
→ 크로스 블로그 발견 가능 (모든 사용자의 #React 글 모음)
```

---

### Step 3: 카테고리 시스템 (2주)

**DB:**
- `categories` 테이블 (blog_id, name, parent_id, order)

**BE:**
- POST/PUT/DELETE `/api/categories` CRUD
- GET `/api/users/:username/categories` 블로그 카테고리 목록

**FE:**
- 에디터 "카테고리 드롭다운"
- 블로그 홈 사이드바 "카테고리 목록"
- `/blog/:username/category/:cat-id` 라우팅

**사용자 가치:**
```
"내 글을 분류할 수 있다"
→ 자신의 블로그 내에서 주제별 정리 가능
```

---

### Step 4: 블로그 홈 완성 (1주)

**FE:**
- 프로필 카드 (이름, 소개, 구독 버튼)
- 태그 클라우드 (자주 사용된 태그 강조)
- 블로그 통계 (포스트 수, 조회수, 댓글 수)

**사용자 가치:**
```
"완성된 블로그 플랫폼"
→ "진짜 블로그 플랫폼이다" 느낌 형성
```

---

### 📅 타임라인

```
주차 1:
  BE: GET /api/users/:username, GET /api/posts?author=
  FE: `/blog/:username` 페이지 기초
  → 멀티블로그 정체성 확립

주차 2-3:
  DB: tags, post_tags 테이블
  BE: 태그 CRUD API
  FE: 에디터 태그 입력, 포스트 상세 태그 표시
  → 발견 기능 추가

주차 4-5:
  DB: categories 테이블 (parent_id 포함)
  BE: 카테고리 CRUD API
  FE: 에디터 카테고리 드롭다운, 블로그 홈 사이드바
  → 정리 기능 추가

주차 6:
  FE: 블로그 홈 프로필 카드, 태그 클라우드, 통계
  → 플랫폼 정체성 완성
```

## 3. 카테고리 vs 태그 UX 설계

### 시각적 구분

**블로그 홈 사이드바:**
```
📁 카테고리              🏷️ 태그 클라우드
├─ 기술 (12)           #react(5)
├─ 일상 (5)            #python(4)
├─ 여행 (3)            #javascript(3)
└─ 책 (2)              ...
```

### 사용자 입장에서의 차이

| 요소 | 카테고리 | 태그 |
|------|---------|------|
| **범위** | 한 블로그 내에서만 유효 | 모든 블로그를 가로지름 |
| **목적** | 정리 (작성자가 분류) | 발견 (사용자가 탐색) |
| **아이콘** | 📁 폴더 | 🏷️ 라벨 |
| **클릭 동작** | `/blog/:username/category/:cat-id` | `/posts?tag=react` |
| **사용자 느낌** | "블로그가 정갈하다" | "관심사별 글을 찾을 수 있다" |

### 온보딩 텍스트

**카테고리:**
```
📁 카테고리: 당신이 글을 체계적으로 정리한 기준입니다.
예) '기술', '일상', '여행'
→ 한 블로그의 카테고리이므로, 당신의 블로그만 특화됨
```

**태그:**
```
🏷️ 태그: 글의 주제를 표시하는 자유로운 라벨입니다.
예) '#React', '#일상', '#여행기'
→ 모든 블로그의 글을 태그로 모아볼 수 있어서 더 많은 글을 발견할 수 있습니다
```

## 4. 블로그 홈 (`/blog/:username`) 필수 요소

## 5. 블로그 커스터마이제이션 UX 개선 (Phase 2.5 ~ 3.1)

> **목적:** "포스트 나열"에서 "내 블로그"로의 전환
> **기대 효과:** 사용자 만족도 ↑ 40%, 재방문율 ↑ (자신의 블로그를 보여주고 싶은 욕구)

### Phase 2.5 (1-2주) — 초기 개인화

**User 모델 확장:**
```python
bio: str (최대 200자)  # 블로그 소개글
avatar_url: str        # 프로필 사진 URL
website_url: str       # 웹사이트 링크
social_links: JSON     # SNS 링크 (GitHub, Twitter, LinkedIn)
blog_title: str        # 커스텀 제목 (기본값: username)
blog_color: str        # 대표 색상 HEX (기본값: --accent)
```

**구현:**
- `/my-blog/settings` 페이지 추가
- "📋 기본 정보" 탭 (프로필 사진, bio, 웹사이트, SNS)
- "🎨 디자인" 탭 (배너 색상, 대표 색상, 레이아웃)
- 우측 실시간 미리보기 패널

**사용자 가치:**
```
"내 블로그가 내 얼굴이다"
├─ 프로필 사진 ← 신뢰도 ↑
├─ 블로그 소개글 ← 개성 표현
├─ 배너 색상 ← 시각적 임팩트
└─ SNS 링크 ← 팔로우 유도
```

---

### Phase 3.1 (2-3주) — 고급 커스터마이제이션

**User 모델 추가:**
```python
banner_image_url: str      # 배너 이미지 URL
blog_layout: str           # 레이아웃 선택 (기본값: 'default')
```

**블로그 레이아웃 옵션:**
- Layout A: 기본 (좌측 사이드바, 카테고리/태그 표시)
- Layout B: 콤팩트 (사이드바 숨김, 포스트 목록만)
- Layout C: 포토그래피 (큼직한 썸네일 그리드)
- Layout D: 매거진 (카드형 메인 포스트 + 리스트)

**사용자 가치:**
```
"내 콘텐츠에 맞춘 블로그 레이아웃"
├─ 배너 이미지 ← Visual Impact
├─ 레이아웃 선택 ← 다양성
└─ 폰트 설정 ← 세밀한 조정
```

## 8. 주의사항 및 함정

### ✅ 해야 할 것

1. **블로그 홈은 '기초 인프라'부터 시작할 것**
   - 카테고리/태그 없이도 블로그 페이지 존재
   - 사용자가 아무 데이터 없어도 방문 가능

2. **태그를 카테고리보다 먼저 구현할 것**
   - 태그: 크로스 블로그 발견
   - 카테고리: 자신의 블로그 관리용

3. **실시간 미리보기**
   - 설정 → 저장 → 블로그 방문 → 확인 (번거로움 ❌)
   - 설정하면서 바로 결과 확인 (WYSIWYG) ✅

4. **온보딩은 매주 새로운 가치를 느끼게**
   ```
   1주 후: "오, 이제 여러 블로그를 볼 수 있네?"
   2주 후: "내가 쓴 '#React' 글들을 한곳에서 보니 좋네!"
   3주 후: "내 블로그를 '기술' '일상' 이렇게 분류했어"
   4주 후: "우와, 진짜 블로그 플랫폼 느낌이네!"
   ```

### ❌ 피해야 할 것

1. ❌ "카테고리를 먼저 구현" → 사용자 가치 40% 이하
2. ❌ "태그와 카테고리를 동시에" → 개발 기간 2배
3. ❌ "블로그 홈을 마지막에" → 모든 기능이 "내 글 편집" 중심
4. ❌ "필드 너무 많음" → 처음 사용자는 overwhelm됨
5. ❌ "색상/폰트 무제한 선택" → 수렴되지 않는 디자인
