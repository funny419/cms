# 🚀 CMS Project Guide (Gemini Edition)

## 프로젝트 개요 및 구조 (Architecture)
본 프로젝트는 **React(FE) + Flask(BE) + MariaDB(DB)** 기반의 설치형 CMS(WordPress 스타일)입니다. 모든 소스는 물리적으로 분리되어 있으며 Docker 컨테이너로 관리됩니다.

---

## Common Commands

### 🖥 Mac 로컬 개발 (docker compose v2)
- **최초 시작**: `docker compose up -d --build`
- **Watch 모드** (파일 변경 자동 반영): `docker compose watch`
- **중지**: `docker compose down`
- **특정 서비스 재시작**: `docker compose restart backend`
- **로그 확인**: `docker compose logs -f`
- **DB Shell**: `docker compose exec db mariadb -u funnycms -p`

### 🗄 DB 마이그레이션
- **스키마 변경 후 파일 생성**: `docker compose exec backend flask db migrate -m "설명"`
- **마이그레이션 적용** (앱 재시작 시 자동): `docker compose exec backend flask db upgrade`

### 🚀 프로덕션 (서버에서 직접)
- **배포**: `docker compose -f docker-compose.prod.yml up -d --build`
- **중지**: `docker compose -f docker-compose.prod.yml down --remove-orphans`

## 🛠 Troubleshooting

### Docker 빌드 오류
- **Nginx Target Failure**: `npm run build` 실패 시 반드시 로컬에서 먼저 빌드하여 문법 오류 수정.
- **SSL 인증서 오류** (`SELF_SIGNED_CERT_IN_CHAIN`): 개발용 `frontend/Dockerfile`에 `npm config set strict-ssl false` 적용되어 있음.
- **`npm ci` 실패** (lock 불일치): 개발용은 `npm install` 사용. `Dockerfile.prod`만 `npm ci` 사용.
- **Gunicorn `Exception in worker process`**:
  1. `app:create_app()` 팩토리 호출 문법 확인.
  2. `docker compose exec backend python app.py`로 ImportError 직접 확인.
  3. `.env` 파일이 프로젝트 루트에 존재하는지 확인.

### Vite / API 연결 오류
- **API 호출 실패 (Docker 내부)**: `vite.config.js`의 proxy target이 `BACKEND_URL` 환경변수로 분기됨.
  - Docker 실행 시 → `http://backend:5000` (docker-compose.yml에서 주입)
  - 로컬 직접 실행 시 → `http://localhost:5000` (기본값)
- **테이블 없음 오류**: `docker compose exec backend flask db upgrade` 실행. 앱 시작 시 자동 적용되도록 설정되어 있음.

## 🚀 CI/CD Maintenance (Windows Runner)
- **Encoding Rule**: PowerShell에서 `.env` 생성 시 반드시 `UTF8 (No BOM)`을 준수할 것.
- **Discord Webhook**: 알림 발송 시 PowerShell 세션 인코딩을 UTF8로 설정하여 한글 깨짐 방지.
- **Cleanup**: 배포 완료 후 `.env` 파일을 삭제하는 `Clean up` 단계가 반드시 `always()` 조건에서 실행되도록 유지할 것.

### 📂 Backend Structure & Imports
- **Import Rule**: 모든 내부 모듈 임포트 시 최상위 폴더(`backend.`)를 생략할 것. 
  - (Docker Build 시 `backend/` 내부 파일이 `/app` 루트로 복사되기 때문)
- **Gunicorn Entry**: `app.py` 최하단에 `app = create_app()`을 선언하여 WSGI 서버가 즉시 인스턴스를 참조할 수 있도록 함.

### 📦 Backend Dependency Rule
- **Baseline**: `requirements.txt`는 Docker 컨테이너의 '법전'임. 
- **Mandatory**: `python-dotenv`, `gunicorn`, `cryptography`를 누락하지 말 것.
- **Sync**: 로컬에서 새로운 패키지를 `import` 했다면, 즉시 `requirements.txt`에 반영하고 `--no-cache` 빌드를 수행할 것.

### 🏗️ Frontend Build Integrity
- **Library Check**: 신규 라이브러리 추가 후 반드시 `package.json` 등록 여부를 `git status`로 교차 검증할 것.
- **Cache Invalidation**: 라이브러리 관련 에러(`Failed to resolve import`) 발생 시, `docker compose build --no-cache`를 통해 클린 빌드를 수행할 것.

### 🌐 Internationalization & Encoding
- **Character Set**: 프로젝트의 모든 소스 코드, 로그, 알림은 **UTF-8**을 원칙으로 함.
- **Windows Deployment**: Windows 서버에서 PowerShell 스크립트 실행 시 `[Console]::OutputEncoding = [System.Text.Encoding]::UTF8`를 선행하여 한글 깨짐 방지.

### 🌐 Windows Runner Encoding - The Gold Standard
- **Strategy**: 스크립트 본문(run 섹션)에는 절대 한글을 직접 작성하지 말 것.
- **Method**: 모든 한글 텍스트는 `env` 섹션을 통해 환경 변수로 전달하고, PowerShell 내에서 `$env:VAR_NAME` 형태로 호출하여 사용할 것.
- **Discord**: JSON 페이로드는 수동으로 조립하되, `UTF8Encoding($false)`를 통한 바이너리 전송 방식을 고수할 것.

### 🐳 Docker Compose Watch (Mac 로컬 개발)
- Mac 로컬에서는 `docker compose watch` 로 파일 변경 자동 감지 및 반영.
- `sync` 액션: Python/JS 파일 변경 → 컨테이너에 즉시 동기화 (Flask auto-reload, Vite HMR 트리거).
- `rebuild` 액션: `requirements.txt` 또는 `package.json` 변경 → 이미지 자동 재빌드.

### 🗄 DB 마이그레이션 워크플로우
- `models/schema.py` 수정 → `flask db migrate -m "설명"` → 앱 재시작 (자동 `upgrade` 실행).
- `create_app()` 내부에서 `flask_migrate.upgrade()` 호출로 앱 시작 시 자동 적용.
- 마이그레이션 파일(`backend/migrations/versions/`)은 반드시 git에 커밋할 것.

## 📝 Coding Standards (Strict Rules)
Karpathy 스타일의 엄격한 가이드라인을 준수할 것.

### General
- **Think First**: 복잡한 로직은 구현 전 `<thought>` 블록에서 설계안을 먼저 제시할 것.
- **Brief & Direct**: 설명은 짧게, 코드는 명확하게 제공할 것. 불필요한 서술 금지.
- **No Hallucination**: 모르는 라이브러리나 버전은 반드시 물어볼 것.

### Backend (Python/Flask)
- **Style**: Flask Blueprints를 사용하여 기능별로 모듈화할 것.
- **Typing**: 모든 함수 선언에 **Python Type Hints** 필수 적용.
- **Response**: 모든 API 응답은 `{ "success": bool, "data": {}, "error": str }` 포맷 준수.
- **Driver**: SQLAlchemy 연동 시 반드시 `pymysql` 드라이버 사용.
- **Import Rule**: 모든 내부 모듈 임포트 시 최상위 폴더(`backend.`)를 생략할 것. 
  - (Docker Build 시 `backend/` 내부 파일이 `/app` 루트로 복사되기 때문)
- **Gunicorn Entry**: `app.py` 최하단에 `app = create_app()`을 선언하여 WSGI 서버가 즉시 인스턴스를 참조할 수 있도록 함.

### Frontend (React/Vite)
- **Style**: Functional Components + **CSS Variables** (`var(--text)`, `var(--bg)` 등). **Tailwind CSS 미설치** — inline style 또는 CSS 변수/유틸리티 클래스 사용.
- **Theme**: Notion/Bear 디자인 시스템 적용 (`index.css`). `useTheme()` 훅으로 다크/라이트 전환. `data-theme="dark"` 속성이 `document.documentElement`에 토글됨.
- **Structure**: API 호출은 `frontend/src/api/` 폴더 내 모듈로 분리. **axios 사용** (fetch 금지).
- **Components**: 공통 컴포넌트는 `frontend/src/components/` 에 위치. Nav 컴포넌트 포함.
- **Proxy**: Vite dev 서버의 `/api` 프록시 target은 `BACKEND_URL` 환경변수로 분기 (`vite.config.js`).
